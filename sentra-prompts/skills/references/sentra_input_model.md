# Sentra Input Model Reference

Canonical input envelope:

<sentra-message-time>
  <time>2026年3月3日 下午 14:34</time>
  <timestamp_ms>1772519640000</timestamp_ms>
  <root>距上次群聊回复过了47分钟，先简要承接上下文再继续。</root>
</sentra-message-time>
<sentra-input>
  <current_messages>
    <sentra-message>
      <chat_type>group|private</chat_type>
      <group_id>... (group only)</group_id>
      <user_id>... (private only)</user_id>
      <sender_id>...</sender_id>
      <sender_name>...</sender_name>
      <message>
        <segment index="1">
          <type>text</type>
          <data>
            <message_id>... (runtime delivery id, optional)</message_id>
            <text>...</text>
          </data>
        </segment>
      </message>
    </sentra-message>
  </current_messages>
  <sentra-pending-messages>...</sentra-pending-messages>
  <sentra-history-messages>...</sentra-history-messages>
  <sentra-tool-results>...</sentra-tool-results>
</sentra-input>

`sentra-message-time` notes:
- Optional runtime metadata block, prepended before `<sentra-input>`.
- `<time>` is local human-readable display text.
- `<timestamp_ms>` is canonical machine ordering timestamp.
- `<root>` (optional) is a short temporal hint generated by runtime when idle gap is large.
- It is not user intent and must not be answered directly.

Priority guidance:
1. `current_messages/sentra-message`
2. current round `sentra-result` blocks
3. pending/history/tool-results
4. summary/objective mirrors
5. memory/RAG background

Callback-round packaging note:
- Runtime may append `<sentra-result>` or `<sentra-result-group>` directly after `</sentra-input>` in the same user payload.
- Treat this as one logical callback input:
  - route anchor: `sentra-input/current_messages/sentra-message`
  - result evidence: appended result blocks
