# 规划完整性（Plan Completeness）

## 目标
- 让计划尽量覆盖用户目标的关键步骤，避免“只做了一半”
- 优先产出可执行的最小闭环，再逐步补充可选步骤

## 方法
1. 先写出 1-2 句“结果定义”（完成后用户应该看到什么）
2. 把任务拆成：
   - 获取信息 / 校验前提
   - 执行动作
   - 验证结果 / 总结
3. 若存在依赖（需要先拿到 id/token/path），必须把“获取依赖信息”作为前置步骤
4. 若工具 schema 的必填字段依赖历史输出，必须在 reason 中明确指出从哪一步的输出提取

## 常见遗漏清单（按需）
- 权限/租户/账号上下文是否需要明确？
- 是否需要先搜索/列举再进行修改/删除/发送？
- 是否需要对结果做校验（例如返回 success=false 时要补救）
- 是否需要给用户一个可追踪的结果（id、链接、输出文件路径）

## 输出偏好
- 计划步骤不宜过多（优先 3-8 步）
- 每一步只做一个明确动作，参数尽量来自上下文，不要猜测
